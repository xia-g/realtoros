# Database Migrations Guide

## Overview

Alembic-based schema migration system for Real Estate OS. All models live in `backend/models/`. Migrations are in `backend/migrations/versions/`.

## File Structure

```
backend/migrations/
├── env.py                 # Alembic environment (async, autogenerate)
├── script.py.mako         # Migration template
├── make_migration.sh      # Helper script
├── README                 # Alembic's built-in README
└── versions/
    └── 2026_06_07_001_001_initial_schema.py  # Initial migration
```

## Quick Start

```bash
# Apply all migrations
source venv/bin/activate
alembic upgrade head

# Or use the helper
./backend/migrations/make_migration.sh --upgrade
```

## Naming Convention

Migrations follow the pattern:

```
YYYY_MM_DD_HHMM_rev_slug.py
```

Example: `2026_06_07_001_001_initial_schema.py`

- **YYYY_MM_DD_HHMM** — date and time of creation
- **rev** — 6-character Alembic revision hash
- **slug** — human-readable description (truncated to 40 chars)

## Versioning Strategy

- **Linear only** — no branching. Each revision has at most one parent and one child.
- **One migration per logical change** — don't combine unrelated schema changes.
- **Never edit applied migrations** — create new ones to fix or extend.
- **Down-revision is always safe** — `alembic downgrade -1` must cleanly revert.

## Commands

### Autogenerate (requires database connection)

Detect model changes and generate a new migration:

```bash
alembic revision --autogenerate -m "description"
```

The helper wraps this:

```bash
./backend/migrations/make_migration.sh "description"
```

### Apply migrations

```bash
alembic upgrade head                  # all pending
alembic upgrade +2                    # next 2
alembic upgrade 001_initial_schema    # up to specific revision
```

### Rollback

```bash
alembic downgrade -1                       # rollback 1 step
alembic downgrade abc123def456             # rollback to specific revision
alembic downgrade base                     # rollback to empty state
```

### History

```bash
alembic history               # show all migrations
alembic current                # show current revision
alembic heads                  # show latest revision (should be 1)
```

## Adding a New Migration

1. **Modify the models** in `backend/models/`
2. **Generate the migration**:
   ```bash
   source venv/bin/activate
   alembic revision --autogenerate -m "add_field_x_to_deals"
   ```
3. **Review the output** in `backend/migrations/versions/` — verify upgrade AND downgrade
4. **Apply**:
   ```bash
   alembic upgrade head
   ```

### Offline mode (no database connection)

```bash
alembic revision --autogenerate -m "description" --sql
```

This generates raw SQL without connecting to the database. Review and apply manually.

## Manual Migration Example

Sometimes autogenerate misses complex changes. Create manual revisions:

```bash
alembic revision -m "add_payment_schedule_to_deals"
```

Then edit the generated file with `op.add_column()`, `op.alter_column()`, etc.

## Best Practices

1. **Always review autogenerated migrations** — autogenerate can miss or misdetect changes
2. **Run `alembic upgrade head` before generating** — ensures clean state
3. **Test downgrade** — `alembic upgrade head && alembic downgrade -1 && alembic upgrade head`
4. **Commit migration files** — they're part of the codebase
5. **Do NOT edit applied migrations** — create new ones to fix mistakes
6. **Keep migration names descriptive** — `add_price_per_meter_to_properties` not `fix_models`

## Troubleshooting

| Problem | Solution |
|---------|----------|
| `NOT NULL violation` on upgrade | Check that new columns with `nullable=False` have `server_default` |
| `Target database is not up to date` | Run `alembic upgrade head` |
| `Can't locate revision` | Check `down_revision` in the new migration file |
| Autogenerate doesn't detect changes | Re-run with `--sql` flag to see generated SQL |
