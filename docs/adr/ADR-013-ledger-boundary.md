# ADR-013: Ledger Boundary — Input Isolation

**Дата:** 2026-06-15
**Статус:** Принято
**Версия:** 1.0.0

---

## Контекст

Завершены Phases 1–2.5. Следующий этап — Ledger (Phase 3).

До реализации необходимо зафиксировать архитектурные границы.

Главный риск: Ledger можно спроектировать с прямым доступом к документам, банковским транзакциям или CRM-событиям, что нарушит воспроизводимость и изоляцию bounded context'ов.

---

## Решение

Ledger принимает ТОЛЬКО `accounting_decision`.

Формула:

```
Posting = f(Decision, PostingRulesVersion)
```

### Аргументы

1. **Воспроизводимость**: Decision уже содержит результат применения правил к snapshot. Ledger не должен повторять эту работу.

2. **Snapshot invariant**: Decision уже изолирован от живых данных. Ledger наследует эту изоляцию.

3. **Версионирование**: Decision имеет `decision_version` и `ruleset_version`. Ledger добавляет только `posting_rules_version`.

4. **Трассировка**: Цепочка `Event → Snapshot → Decision → Posting` полностью замкнута.

---

## Последствия

### Positive

- Ledger не зависит от схемы CRM, документов или банков
- Replay изолирован: пересчёт решения → новая проводка, старая не меняется
- Чёткая граница ответственности: Decision Engine решает «что», Posting Engine — «как разнести»
- Period lock прозрачен: закрытый период блокирует posting, но не decision

### Negative

- Posting Engine должен понимать семантику `decision_state`: INCLUDED → posting, EXCLUDED → нет
- Нужен дополнительный слой `posting_decision_link` для трассировки
- Если posting rules зависят от суммы/типа события (что нормально), эти данные уже есть в accounting_decision

---

## Отклонённые варианты

### Вариант A: Posting = f(Event)

Проводки строятся напрямую из accounting_event.

**Отклонено:** нарушает архитектуру. Decision Engine существует именно для того, чтобы отделить «что произошло» от «как учесть». Event может быть INCLUDED или EXCLUDED — Ledger не должен знать об этом выборе.

### Вариант B: Posting = f(Document)

Ledger читает документы напрямую.

**Отклонено:** полная потеря воспроизводимости. Документ может измениться (OCR перезапущен, распознавание улучшено). Ledger должен зависеть только от того, что уже зафиксировано.

### Вариант C: Posting = f(Decision, Event)

Гибридный подход — решение определяет «включать или нет», событие даёт сумму.

**Отклонено:** дублирование входа. Все необходимые данные (сумма, контрагент, тип) уже есть в decision (и, через snapshot, в accounting_event).
