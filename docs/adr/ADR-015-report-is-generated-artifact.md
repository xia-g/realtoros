# ADR-015: Report — Generated Artifact, Not Business Logic

**Дата:** 2026-06-16
**Статус:** Проект
**Версия:** 1.0.0

---

## Контекст

Phase 4 (Tax Registers) завершён. Следующий этап — генерация регламентированной отчётности.

Главные вопросы:

1. **Report = f(что)?** Должен ли отчёт напрямую читать Ledger, или только TaxRegister?
2. **Где хранить бизнес-логику?** Должен ли Report содержать формулы расчёта?
3. **Как быть с шаблонами?** Хранить их как код, как данные, или загружать извне?
4. **Кто может менять отчёт?** AI? Бухгалтер? Автоматическая корректировка?

---

## Решение

### 1. Report = f(TaxRegister, ReportTemplateVersion)

Отчёт строится исключительно на данных налогового регистра и версии шаблона.

Никаких других входов.

**Почему:**
- TaxRegister — единственный авторитетный источник налоговых данных
- TemplateVersion — единственный авторитетный источник формы
- Всё остальное (Ledger, Decision, Event) — обход слоёв, ведущий к потере объяснимости

### 2. Report — сгенерированный артефакт

Report не хранит:
- Формулы расчёта
- Логику валидации
- Правила конвертации
- AI модели

Report хранит:
- Данные (суммы, ИНН, КПП, период)
- Ссылки (template_version, register_version, policy_version, generated_at)

**Почему:**
- Если формула меняется → новая template_version → перегенерация
- Если валидация меняется → старые отчёты не пересчитываются
- Report — слепок, не интерпретатор

### 3. Template — внешний артефакт

Template загружается из официальных источников (nalog.ru, minfin.ru).

- immutable: новая версия = новый template_version
- checksum: защита от подмены
- effective_from / effective_to: автоматическая миграция

**Нельзя редактировать шаблон вручную.**

### 4. Audit — независимая интерпретация

AI Audit не меняет Report.

AuditResult прикладывается к отчёту, но не модифицирует его.

**Почему:**
- Audit — это мнение, Report — это факт
- AI может ошибаться (галлюцинации)
- Юридическая ответственность лежит на подписанте

### 5. Отчёт immutable после signing

После подписания ЭЦП отчёт не может быть изменён.

Коррекция = новая версия отчёта (report_v1 → report_v2).

---

## Последствия

### Positive

- Чёткая граница между данными (Register) и формой (Template)
- Report можно пересобрать в любой момент (template + register)
- Audit можно перезапустить без изменения отчёта
- Template меняется независимо от pipeline
- Каждая цифра объяснима (chain до decision_explanation)

### Negative

- Дополнительный JOIN для связки report → template → register → assignment
- Нужна инфраструктура для загрузки и версионирования шаблонов
- Нельзя быстро «подправить цифру» в отчёте — только новая версия
- AI Audit требует отдельного раннера (не в pipeline)

---

## Отклонённые варианты

### Вариант A: Report = f(Ledger)

Report напрямую читает Ledger и строит отчёт.

**Отклонено.** Нарушает Layer Isolation. Если налоговая политика меняется, Ledger не перестраивается, но Report должен измениться — несоответствие слоёв. Потеря объяснимости: ячейка отчёта ссылается на ledger_line, но не на tax_treatment.

### Вариант B: Report хранит копию формул из Template

При генерации Report копирует формулы из Template и хранит их как часть данных отчёта.

**Отклонено.** Раздувает модель данных. Дважды хранит одну и ту же логику. При смене Template старые отчёты содержат устаревшие формулы — путаница. Report перестаёт быть «данными» и становится «кодом».

### Вариант C: AI Audit может исправлять Report

AI не только находит ошибки, но и автоматически корректирует цифры в отчёте.

**Отклонено.** Юридически недопустимо (ст. 120 НК РФ — ответственность на подписанте). AI галлюцинации приводят к неверным корректировкам. Теряется traceability: непонятно, кто изменил цифру — AI или человек. Нарушает immutable invariant.

### Вариант D: Отчёт = XML-файл, без модели

Report хранится только как сырой XML/Excel без структурированной модели.

**Отклонено.** Невозможно построить explainability chain. Невозможно версионировать. Невозможно сравнивать версии (diff). Нельзя ответить на вопрос «почему эта цифра?».

### Вариант E: Одна template_version на все режимы

Универсальный шаблон, который подстраивается под tax_regime компании.

**Отклонено.** Каждый режим (УСН, ОСНО, Патент) имеет свою форму отчётности с разными полями, разделами и контрольными соотношениями. Универсальный шаблон = сложная логика ветвления = высокий риск ошибки. Лучше n простых шаблонов, чем 1 сложный.

### Вариант F: Report как source of truth

Report считается источником истины для налоговых данных. Все остальные слои (register, assignment) подстраиваются под report.

**Отклонено.** Report — производная величина, не первичная. Если report source of truth, любое исправление отчёта требует пересчёта register, assignment, ledger — что нарушает layer isolation. Report должен быть пересобираем, а Register — источником.

### Вариант G: Template embedded into report

При генерации шаблон встраивается в отчёт (полная копия полей, формул, контрольных соотношений).

**Отклонено.** Дублирует данные. При смене шаблона (например, из-за изменения законодательства) старые отчёты содержат устаревшие шаблоны — невозможно определить, по какой версии считается каждая ячейка. Report перестаёт быть «слепком» и становится «монолитом».

### Вариант H: Audit mutates report

AI Audit имеет право исправлять цифры в отчёте (автоматическая корректировка).

**Отклонено.** Юридически недопустимо — подпись ставит человек. AI галлюцинации ведут к неверным корректировкам. Теряется immutable invariant (кто изменил? AI или человек?). Нарушает explainability (цепочка обрывается на AI). Дублирует решение: если AI может исправлять, кто проверяет AI?

### Вариант I: Submission mutates report

Отправка отчёта в ФНС меняет его статус внутри системы (например, rejected → корректировка существующего report).

**Отклонено.** Submission — транспорт, не владелец. Статус ФНС влияет только на submission, не на report. Если report rejected → новая версия отчёта, не правка старой. Иначе submission становится source of truth для отчётных данных.
